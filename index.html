01

Software Engineering Fundamentals

Quality Attributes (The Three Pillars)
Reliability CORE
Definition: The system continues to work correctly even in the face of adversity — hardware faults, software bugs, and human errors.

Reliability is measured by how well a system handles failures gracefully without data loss or service interruption.

Failure Sources

  • Hardware: disk crashes, RAM faults, power failures
  • Software: bugs, runaway processes, cascading failures
  • Human: config mistakes, bad deployments, accidental deletions

How to Achieve It

  • Design for failure — assume every component will eventually fail
  • Use redundancy (multiple servers, multiple data centres)
  • Netflix Chaos Monkey: intentionally kills production servers
  • Automated monitoring & recovery
💡
Real World: The 2024 CrowdStrike outage crashed 8.5 million Windows systems worldwide — a single faulty software update caused ~$5.4 billion in Fortune 500 losses. One reliability failure cascaded globally.
Scalability CORE
Definition: As the system grows (data volume, traffic, or complexity), there should be reasonable ways of dealing with that growth.
TypeAlso CalledApproachProsCons
Vertical ScalingScale UpBigger, more powerful machineSimple, no code changesExpensive, has limits, single point of failure
Horizontal ScalingScale OutAdd more machines, distribute loadIndefinite scaling, fault tolerant, cost-effectiveComplex distributed systems

Common Scalability Patterns: Load Balancing, Caching, Database Sharding, Async Processing, CDNs

⚠️
Pokémon GO (2016): Niantic expected normal launch traffic. Social media made it viral instantly. Millions of users flooded servers within hours — login, maps, and Gyms all became bottlenecks. Classic scalability failure.
Maintainability CORE
Definition: The ability of a system to be understood, modified, tested, and extended with low cost and low risk over time.

Most system design problems fail after launch, not at launch. Engineering effort is mostly spent on changes, not greenfield development.

  • Separation of Concerns: Business logic ≠ infrastructure; API ≠ data storage
  • Modularity: Components can change independently
  • Simplicity: Clear > clever; boring solutions age better
Durability CORE
Definition: The ability of a system to preserve data correctly and permanently, even in the presence of crashes, power failures, network partitions, and process restarts.

Once a system acknowledges a write, it should never be lost. Critical for financial systems, user data, logs, events, and transactions.

Software Project Lifecycle
SDLC Phases CORE

Software is built in stages. Skipping any phase creates bigger problems downstream.

Planning → Requirements → Design → Development → Testing → Deployment → Maintenance
PhaseQuestion AnsweredOutput
PlanningIs it worth building?Project plan, risk assessment
RequirementsWhat to build?SRS (Software Requirements Specification)
DesignHow to build it?Architecture diagrams, DB schemas, UML
DevelopmentBuilding itSource code
TestingDoes it work correctly?Test reports, bug fixes
DeploymentHow to release it?Live system accessible to users

🚨
Skipping phases costs you: No requirements → wrong product. No design → poor structure. No testing → bugs in production. No maintenance → dead product.
02

OOP Core Concepts

Classes & Objects
Class — The Blueprint CORE
Definition: A class is a blueprint or template that defines the attributes (data/state) and behaviours (methods/functions) that objects of that type will have.

A class only occupies memory as a template — it is not a real thing until an object is created from it.

TypeScript — Class Definition
class Player {
  name: string;
  health: number;
  currentWeapon: string;
  isAlive: boolean;

  pickWeapon(weapon: string): void { /* ... */ }
  fire(): void              { /* ... */ }
  enterVehicle(): void     { /* ... */ }
}
Object — The Real Instance CORE
Definition: An object is a real instance of a class — it occupies memory, has its own state and identity, and can be interacted with.

Every object has three characteristics:

  • State — what it knows (its attribute values)
  • Behaviour — what it does (its methods)
  • Identity — who it is (its unique reference in memory)
TypeScript — Creating Objects
const player1 = new Player("Vikram", 100);
const player2 = new Player("Riya", 100);
// Both created from the same class, but are different objects
Constructors
Constructor CORE
Definition: A constructor is a special method that runs automatically when an object is created. Its job is to set the initial valid state of the object.
  • If you don't write one, TypeScript/JavaScript provides a default empty constructor: constructor() {}
  • Constructors are the best place to enforce invariants at creation time
  • They ensure no object is ever created in an invalid state
TypeScript — Constructor with Validation
class Car {
  model: string;
  seatingCapacity: number;

  constructor(model: string, seats: number) {
    if (seats <= 0) {
      throw new Error("Seating capacity must be > 0");
    }
    this.model = model;
    this.seatingCapacity = seats;
  }
}
Invariants
Invariants — Rules That Must Always Hold CORE
Definition: An invariant is a condition about an object's state that must always be true — before and after every operation.

Example invariants:

  • A Car must always have at least 1 seat (seatingCapacity > 0)
  • A Player's health must be between 0 and 100
  • A bank account balance must never go below the overdraft limit
✅
Why invariants matter: Consistent and predictable object behaviour. Fewer bugs through early error detection. Better maintainability with clear rules baked into the class.

OOP Mechanisms

Encapsulation
Encapsulation CORE
Definition: Bundling data (attributes) and the methods that operate on that data within one class, while restricting direct external access to the data.

The Rule: Mark instance variables private. Expose them through public getters and setters that enforce invariants.

❌ Without Encapsulation (Naked Object)

class Dog {
  public size: number; // exposed!
}
const fido = new Dog();
fido.size = -50; // valid code, fatal logic

✅ With Encapsulation

class Dog {
  private _size: number;
  set size(v: number) {
    if(v <= 0) throw Error();
    this._size = v;
  }
}

Abstraction
Abstraction CORE
Definition: Hiding complex implementation details and showing only the essential features. Focus on WHAT it does, not HOW it does it.

Like driving a car — you use the steering wheel and pedals (abstraction), without needing to know about fuel injection or the transmission.

abstract class Vehicle {
  abstract getSpeed(): number; // must be implemented
  describe(): void { // concrete (shared) method
    console.log(`Speed: ${this.getSpeed()}`);
  }
}
class Car extends Vehicle {
  getSpeed(): number { return 120; }
}
TypeScript — Interface (Contract)
interface Payable {
  processPayment(amount: number): void;
  validatePayment(): boolean;
}
class CreditCard implements Payable { /* ... */ }
class UPI        implements Payable { /* ... */ }
💡
Rule of Thumb: Need flexibility across unrelated classes → use Interface. Need shared code among related classes → use Abstract Class.
Inheritance
Inheritance (IS-A) CORE
Definition: Allows one class (child) to inherit the properties and methods of another class (parent). Represents an IS-A relationship.

Test: "A Dog IS-A Animal" → ✅ use inheritance. "A Dog IS-A Collar" → ❌ use composition.

TypeScript — Inheritance
class Animal {
  eat()  { console.log("Eating..."); }
  sleep(){ console.log("Sleeping..."); }
}
class Dog extends Animal {
  bark() { console.log("Woof!"); } // extra behaviour
}
const d = new Dog();
d.eat();  // inherited from Animal
d.bark(); // Dog's own method
⚠️
Common Pitfalls: Deep hierarchies (changing base class wrecks everything). Over-generalisation (Penguin IS-A Bird but can't fly — broken promise). Favour Composition over Inheritance for flexibility.
Polymorphism
Polymorphism CORE
Definition: Poly = Many, Morphism = Forms. The ability of objects to take multiple forms — one interface, multiple implementations.
TypeAlso CalledWhen DecidedMechanism

RuntimeMethod OverridingAt executionChild class overrides parent method
TypeScript — Method Overriding (Runtime Polymorphism)
class Animal {
  sound(): string { return "..."; }
}
class Dog extends Animal {
  sound(): string { return "Woof"; }  // overrides
}
class Cat extends Animal {
  sound(): string { return "Meow"; }  // overrides
}
const animals: Animal[] = [new Dog(), new Cat()];
animals.forEach(a => console.log(a.sound())); // Woof, Meow
Code Reusability Flexibility Maintainability Cleaner Code

Object Relationships & Design Concepts

Composition vs Inheritance
Composition (HAS-A) CORE
Definition: A class contains or uses another class as a part. "Dog HAS-A Tail" — the Dog class holds a reference to a Tail object. It does not become one.
Inheritance (IS-A)Composition (HAS-A)
"You are this thing""You have this thing"
Dog extends AnimalDog has a Tail
Child gets parent DNA automaticallyObject borrows another object's skills
One wrong change → many breakChange stays in one place
Feels powerful, gets riskyFeels boring, stays safe
Use when relationship is obvious IS-AUse when relationship feels forced
💡
Design Rule: Favour composition over inheritance for flexibility and safer design. "Has-a" is usually the better choice when in doubt.
DTO vs Domain Object
DTO (Data Transfer Object) vs Domain Object CORE
AspectDTODomain Object
PurposeCarry data between layers (API → Service → DB)Represent core business entity with rules
Has Logic?No — data only (plain fields)Yes — enforces invariants and business rules
ExampleUserDTO { name: string; email: string }User { private _balance; deposit(amt) { ... } }
VisibilityPublic fieldsPrivate fields + public methods
TypeScript — DTO vs Domain Object
// DTO — plain data bag
interface UserDTO {
  name: string;
  email: string;
}

// Domain Object — enforces rules
class BankAccount {
  private _balance: number;
  deposit(amt: number) {
    if (amt <= 0) throw Error("Amount must be positive");
    this._balance += amt;
  }
}

Design Patterns

What Are Design Patterns? PATTERN
Definition: Design patterns are well-proven, reusable solutions to commonly occurring problems in software design. They are templates, not finished designs.
Creational — How objects are created Structural — How objects are composed Behavioral — How objects communicate
Creational Patterns
Singleton Pattern CREATIONAL
Problem: We need exactly one instance of a class, accessible globally throughout the application (e.g., logger, config, DB connection pool).

Three Key Components:

  • Private Constructor — no one can use new to create instances from outside
  • Static Field — holds the single instance, shared across all usage
  • Static Access Method: getInstance() — creates on first call, returns existing instance on subsequent calls (lazy initialization)
TypeScript — Singleton
class Logger {
  private static instance: Logger;
  private constructor() {}  // private — no external new

  static getInstance(): Logger {
    if (!Logger.instance) {
      Logger.instance = new Logger();
    }
    return Logger.instance;
  }
  log(msg: string) { console.log(msg); }
}
const a = Logger.getInstance();
const b = Logger.getInstance();
console.log(a === b); // true — same instance
⚠️
Pitfalls: Acts like a global variable (global state). Cannot mock/replace in unit tests — tests become interdependent. Often considered an anti-pattern by senior engineers.
Factory Pattern CREATIONAL
Problem: Object creation logic becomes complex and scattered. We want to decouple clients from concrete classes and hide which class is actually instantiated.

Structure: Abstraction/Interface → Concrete Products → Simple Factory

TypeScript — Notification Factory
interface Notification { send(msg: string): void; }
class EmailNotification implements Notification {
  send(msg: string) { console.log(`Email: ${msg}`); }
}
class SMSNotification implements Notification {
  send(msg: string) { console.log(`SMS: ${msg}`); }
}

class NotificationFactory {
  static create(type: string): Notification {
    if (type === 'email') return new EmailNotification();
    if (type === 'sms')   return new SMSNotification();
    throw new Error("Unknown type");
  }
}
// Client never knows which class was created
const n = NotificationFactory.create('email');
n.send("Hello!");
Structural Patterns
Adapter Pattern STRUCTURAL
Problem: Your code expects one interface, but a third-party library provides a different one. You cannot modify the third-party code.

Core Principle: Don't change existing code — wrap it instead!

Your System Adapter Third-Party PaymentProcessor ───► RazorpayAdapter ───► ThirdPartyPayment pay(amount) pay(amount) makePayment(usd) [translates call]
TypeScript — Adapter Pattern
// Target Interface (what your system expects)
interface PaymentProcessor {
  pay(amount: number): void;
}
// Adaptee (third-party you can't change)
class ThirdPartyPaymentService {
  makePayment(valueInDollars: number) { /* razorpay logic */ }
}
// Adapter — the translator
class PaymentAdapter implements PaymentProcessor {
  private service = new ThirdPartyPaymentService();
  pay(amount: number) {
    this.service.makePayment(amount / 83); // ₹ → $
  }
}
Composite Pattern STRUCTURAL
Problem: You have tree-like structures (files + folders, org charts) and want to treat individual items and containers the same way.

Core Principle: Treat part and whole uniformly. Client doesn't check "is this a file or folder?"

Component Interface (FileSystemItem: getSize()) │ ├── Leaf (File): has no children, returns own size └── Composite (Folder): holds children, returns sum of all children's sizes
TypeScript — Composite Pattern
interface FileSystemItem {
  getSize(): number;
}
class File implements FileSystemItem {
  constructor(private size: number) {}
  getSize() { return this.size; }  // leaf: just own size
}
class Folder implements FileSystemItem {
  private children: FileSystemItem[] = [];
  add(item: FileSystemItem) { this.children.push(item); }
  getSize() { // composite: sum of all children (recursive)
    return this.children.reduce((sum, c) => sum + c.getSize(), 0);
  }
}
Behavioral Patterns
Observer Pattern BEHAVIORAL
Problem: When one object changes state, many other objects need to react automatically, without tight coupling between them.

Analogy: WhatsApp broadcast — you send one message, everyone in the group receives it automatically.

Subject (Publisher) Observers (Subscribers) TemperatureSensor ──notify()──► DisplayObserver ──notify()──► LoggerObserver ──notify()──► AlertObserver
TypeScript — Observer Pattern
interface Observer { update(data: number): void; }
interface Subject {
  attach(obs: Observer): void;
  detach(obs: Observer): void;
  notify(): void;
}
class TemperatureSensor implements Subject {
  private observers: Observer[] = [];
  private temp: number = 0;
  setTemperature(t: number) { this.temp = t; this.notify(); }
  attach(obs: Observer) { this.observers.push(obs); }
  detach(obs: Observer) { /* remove from array */ }
  notify()  { this.observers.forEach(o => o.update(this.temp)); }
}
⚠️
Common Pitfalls: Memory leaks if observers are never detached. No guaranteed notification order. Performance issues with 10,000+ observers.
Strategy Pattern BEHAVIORAL
Definition: Defines a family of algorithms, encapsulates each one in a separate class, and makes them interchangeable at runtime. The algorithm varies independently from the clients that use it.

Real-World: Google Maps — same app, different route algorithms (Car, Bike, Walking, Public Transport). You can switch modes anytime without changing the app itself.

Context (Checkout) HAS-A Strategy Interface (PaymentMethod) │ ┌───────────────┼───────────────┐ CreditCard UPI PayPal
TypeScript — Strategy Pattern
interface PaymentStrategy { pay(amount: number): void; }

class CreditCardStrategy implements PaymentStrategy {
  pay(amount: number) { console.log(`Credit Card: ₹${amount}`); }
}
class UPIStrategy implements PaymentStrategy {
  pay(amount: number) { console.log(`UPI: ₹${amount}`); }
}

class Checkout {
  private strategy: PaymentStrategy;
  setPaymentMethod(s: PaymentStrategy) { this.strategy = s; }
  processOrder(amount: number) { this.strategy.pay(amount); }
}
// Switch at runtime — Checkout class never changes!
const checkout = new Checkout();
checkout.setPaymentMethod(new UPIStrategy());
checkout.processOrder(1500);
Template Method Pattern BEHAVIORAL
Definition: Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the algorithm's structure.

Mechanism: Inheritance (IS-A) — the parent controls the flow; children fill in the blanks.

AspectStrategy PatternTemplate Method
MechanismComposition (HAS-A)Inheritance (IS-A)
Runtime ChangesPossible — swap strategy objectImpossible — fixed at class definition
Use CaseInterchangeable algorithmsFixed workflow with small variations
ExamplePayment methodsData processing pipeline, Report generation
💡
Rule: Choose Strategy when you need to swap behaviour at runtime. Choose Template Method when you have a fixed process with customisable steps.

SOLID Principles

SOLID is a set of 5 core principles for clean OOP design. They are guidance, not strict rules. Think of them as a cheat sheet to prevent messy, fragile, and rigid code.

S — Single Responsibility O — Open/Closed L — Liskov Substitution I — Interface Segregation D — Dependency Inversion
S — Single Responsibility Principle (SRP) SOLID
Definition: A class should have one and only one reason to change. Responsibility = the actor/stakeholder who would request the change.
⚠️
Common Misconception: SRP ≠ "one method". A class can have many methods but still have ONE reason to change (one owner/stakeholder).
SRP Violation → Fix
// ❌ Violation: Student class handles data + reporting + DB
class Student {
  name: string;
  generateReport() { /* ... */ }  // UX Designer owns this
  saveToDatabase() { /* ... */ }  // DBA owns this
}
// ✅ Fixed: three focused classes
class Student           { name: string; }
class ReportGenerator  { generate(s: Student) { /* ... */ } }
class StudentRepository{ save(s: Student)    { /* ... */ } }
O — Open-Closed Principle (OCP) SOLID
Definition: Software entities should be open for extension but closed for modification. Add new features by writing new code, not by editing stable existing code.

How: Identify the "seam" (variation point) once. Use Strategy Pattern, Interfaces, or Abstract Classes. Future features become new classes, not edits to old ones.

OCP Violation → Fix
// ❌ Violation: Adding Senior discount requires editing this class
class DiscountService {
  getDiscount(type: string) {
    if (type === 'student')  return 0.1;
    if (type === 'employee') return 0.2; // keep adding ifs...
  }
}
// ✅ Fixed: new discounts = new classes, service unchanged
interface DiscountStrategy { getDiscount(): number; }
class StudentDiscount  implements DiscountStrategy { getDiscount() { return 0.1; } }
class SeniorDiscount   implements DiscountStrategy { getDiscount() { return 0.15; } }
class DiscountService  { apply(d: DiscountStrategy) { return d.getDiscount(); } }
L — Liskov Substitution Principle (LSP) SOLID
Definition: Subtypes must be substitutable for their base types without altering the correctness of the program. If code works with a parent class, it must also work with ANY child class — no surprises.

LSP is about behavioural promises, not inheritance syntax.

LSP Violation
class Bird    { fly() { console.log("Flying"); } }
class Ostrich extends Bird {
  fly() { throw new Error("Ostriches can't fly!"); }
}
// ❌ Breaks LSP: Bird promises fly() works, Ostrich breaks the promise
// ✅ Fix: Restructure hierarchy — FlyingBird and NonFlyingBird
I — Interface Segregation Principle (ISP) SOLID
Definition: Clients should not be forced to depend on methods they do not use. Many small, focused interfaces are better than one large "fat" interface.
ISP Violation → Fix
// ❌ Fat Interface — SimplePrinter forced to implement scan() and fax()
interface Machine { print(): void; scan(): void; fax(): void; }

// ✅ ISP Fix — split into focused interfaces
interface Printable { print(): void; }
interface Scannable { scan(): void; }
interface Faxable   { fax(): void; }
class SimplePrinter implements Printable  { /* only print */ }
class AllInOnePrinter implements Printable, Scannable, Faxable {}
D — Dependency Inversion Principle (DIP) SOLID
Definition: High-level modules should not depend on low-level modules. Both should depend on abstractions. Depend on WHAT you want to do, not HOW it is done.
💡
MCQ Answer: Constructor Injection (passing dependencies via constructor) is the most direct implementation of DIP — answer is C (DIP).
DIP Example
// ❌ High-level depends on low-level concrete class
class NotificationService {
  private emailSender = new EmailSender(); // tightly coupled
}
// ✅ Both depend on abstraction
interface MessageSender { send(msg: string): void; }
class NotificationService {
  constructor(private sender: MessageSender) {} // injected
  notify(msg: string) { this.sender.send(msg); }
}
// Now can inject Email, SMS, or Slack — no changes to NotificationService

SDLC Models

SDLC vs Process Model
SDLC (The WHAT)Process Model (The HOW)
The steps that must happen (Requirements, Design, Dev, Test, Deploy, Maintain)The way teams execute those steps (Waterfall = Linear, Agile = Iterative)
Waterfall Model
Waterfall — Linear Sequential Model SDLC
Analogy: Building a house — design everything upfront (blueprints), lay the foundation, build walls. Changes are EXPENSIVE after the foundation is laid.
Requirements ──► Design ──► Development ──► Testing ──► Deployment ──► Maintenance (fixed) (upfront) (one shot) (end only) (one release) (bugs only)

✅ When to Use

  • Requirements are clear and fixed from the start
  • Changes are very unlikely during development
  • Documentation and approvals required at every stage
  • Large, critical, or highly regulated systems

Real Examples

  • Defence systems (MIL-STD-498)
  • Core banking and payroll systems
  • Space programs (NASA Mars missions)
  • Large-scale enterprise infrastructure
Agile Model
Agile — Iterative & Incremental SDLC
Analogy: Cooking a dish — make a simple version, taste it, adjust spices based on feedback, taste again. Continuous feedback and improvement.
Sprint 1: Build simple version → Test → Feedback Sprint 2: Improve based on feedback → Test → Feedback Sprint 3: Add more features → Test → Feedback ↑ Each sprint = 1–4 weeks, produces working software ↑

CI/CD Pipeline
Continuous Integration / Continuous Deployment SDLC

CI — Continuous Integration

Code is automatically tested every time a developer pushes. Catches errors early. No manual testing for routine checks. Fast feedback.

CD — Continuous Deployment

Code automatically deployed through stages (staging → production). Reduces manual work and errors. Code goes live faster.

Developer Writes Code ↓ Push to Git/GitHub (version control) ↓ Code Review (senior dev checks quality) ↓ Automated Tests Run (CI) — catches bugs before merge ↓ Build (code is packaged) ↓ Staging Deployment (test on real servers) ↓ QA Testing ↓ Production Deployment (CD) — goes live to users ↓ Monitor for errors + alerts
🏭
Real Companies: Amazon — code to production in MINUTES. Netflix — thousands of changes daily with instant rollback. Swiggy — deploys new features multiple times per day.

Testing

Why Testing? TESTING

Software assumptions fail in real usage. Users behave unpredictably. Systems grow beyond developer control. Testing validates behaviour before release.

Cost of NOT Testing

  • Bugs reach production
  • User trust is lost
  • Emergency fixes increase cost
  • Teams shift from building → firefighting

Testing = Engineering Discipline

  • Not optional cleanup
  • Not only QA's responsibility
  • Core part of system reliability
  • Enables safe scaling
Unit Testing & TDD
Unit Testing TESTING
Definition: Tests the smallest testable logic in isolation — individual functions or methods. Validates expected behaviour. Fast and isolated.

"Unit tests are like saving checkpoints — when you change something, they tell you what you accidentally broke."

  • Catch logic bugs early
  • Prevent regression (old features breaking)
  • Enable confident refactoring
  • Improve code quality through testable design
TypeScript — Jest Unit Test
// Function to test
function calculateFinalPrice(price: number, discount: number) {
  if (discount < 0 || discount > 1) throw new Error("Invalid");
  return Math.max(0, price * (1 - discount));
}
// Jest tests
describe('calculateFinalPrice', () => {
  it('applies 10% discount correctly', () => {
    expect(calculateFinalPrice(1000, 0.1)).toBe(900);
  });
  it('never returns negative price', () => {
    expect(calculateFinalPrice(100, 1.5)).toThrow();
  });
});
Test-Driven Development (TDD) TESTING
Definition: Write the test BEFORE writing the code. Forces clear requirements and improves design clarity. "TDD is deciding what 'correct' means before coding."
🔴 RED → Write a failing test (the feature doesn't exist yet) 🟢 GREEN → Write the minimum code to make the test pass 🔵 REFACTOR → Clean up the code while keeping tests green Repeat for every small feature unit
Integration Testing
Integration Testing TESTING
Definition: Tests interactions between components — validates APIs, databases, and services working together correctly. Sits above unit tests in the testing pyramid.

Why Integration Tests Fail: Incorrect data flow, API contract mismatch, environment differences, dependency issues.

AspectUnit TestingIntegration Testing
FocusLogic correctness of one functionComponent collaboration (A calls B correctly)
IsolationFully isolated (mocks)Real or near-real dependencies
SpeedVery fast (milliseconds)Slower (seconds)
ToolsJest, Mocha, JUnit, pytestPostman, Supertest
💡
Key Insight: Unit tests and integration tests are complementary — one cannot replace the other. Both are required for system stability.
End-to-End (E2E) Testing
E2E Testing TESTING
Definition: Tests the system from the user's point of view — covering the entire flow from UI through backend to database and external services.
  • What it verifies: User can complete a task start to end; all services work together; correct data is saved; user sees the expected result
  • What it catches that others don't: Broken user flows, configuration mistakes, deployment issues, missing service connections
  • Why it's expensive: Involves real systems, requires databases + servers + UI rendering, failures are harder to trace
⚖️
Testing Pyramid: Many Unit Tests (fast, cheap) → Fewer Integration Tests → Very Few E2E Tests (slow, expensive). Don't invert the pyramid.
